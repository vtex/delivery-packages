# @vtex/delivery-packages Constitution

This document records the **permanent, non-negotiable** rules that govern changes to this repository. Anything that is merely a current preference or a temporary command belongs in `README.md` or `AGENTS.md`, not here.

## Library posture

This repo is **a published JavaScript library** on npm (`@vtex/delivery-packages`). Every change to the public API affects every consumer. The default disposition for any change is:

- Smallest correct change
- Additive over breaking
- Documented with tests, CHANGELOG, and (for breaking changes) cross-consumer review

## Stack and runtime

- **Build system:** Rollup 0.56 driven by `rollup.config.js`. Produces `dist/index.js` (CJS) and `dist/index.esm.js` (ESM). No alternative bundler (Webpack, Vite, esbuild, tsup) may be introduced without a formal migration ADR
- **Transpilation:** Babel 6 with `babel-preset-env` (production sets `modules: false` so Rollup emits ESM). `babel-plugin-transform-object-rest-spread` and `external-helpers` are in the chain
- **Source language:** JavaScript only — pure ES modules using `import`/`export`. No TypeScript at the time of writing; wholesale conversion requires a coordinated migration plan
- **Testing:** Jest 22 + babel-jest 22
- **Linting:** ESLint 4 + `eslint-config-vtex` + `eslint-plugin-jest`
- **Formatting:** Prettier (`.prettierrc`: no semicolons, single quotes, trailing comma es5)
- **Git hooks:** Husky 1 (`lint-not-beta`, `test-not-beta` skip on the `beta` release-staging branch)
- **Package manager:** both `yarn.lock` and `package-lock.json` are present. Yarn is the historical choice; npm works. `packageManager` is not pinned in `package.json`

## Public API contract

- **`src/index.js` defines the public API.** Every named export from `src/index.js` is a contract with downstream consumers
- Parameter signatures (names, order, types), return shapes (field names, optionality), and default values are part of the contract
- Adding a new optional parameter at the end is non-breaking; renaming, reordering, or changing defaults is breaking
- Breaking changes require:
  - `BREAKING CHANGE:` footer in the Conventional Commit
  - Explicit `### Breaking Changes` entry in `CHANGELOG.md`
  - Cross-consumer impact analysis in the spec (when SDD applies)
  - Coordination with downstream consumers — agents do not make this call alone

## Dependency rules

- **Runtime dependencies (`dependencies`)** ship to every consumer. The current single runtime dep is `@vtex/estimate-calculator`. Adding another requires explicit justification in the PR
- **devDependencies** are pinned to old majors (Babel 6, Jest 22, ESLint 4, Rollup 0.56). Bumping is a coordinated tooling decision
- **Peer dependencies** must not be added without team approval — they complicate consumer installs
- **Forbidden categories:** anything that hard-requires Webpack 4+, anything that mandates ESM-only at consume time, anything that pulls in a transitive runtime dependency that is itself heavy

## Architectural rules

- **Source files in `src/` are pure helper modules.** No side effects on import. No DOM access. No network calls. No storage access
- **`src/index.js` is the public surface.** Internal-only helpers must not be re-exported from it
- **`src/constants.js`** is the place for shared enums. Do not duplicate string literals across modules
- **`src/polyfills.js`** is intentionally minimal — only polyfills strictly required by the helpers
- **`tests/` mirrors `src/`.** Test files are named `<name>.test.js`. `tests/Order.json` is the shared fixture; `tests/mockGenerator.js` is the shared test helper
- **`dist/`** is generated by Rollup at publish time — never committed, never edited

## Code standards

- New code is JavaScript. TypeScript may be considered as a coordinated migration but is not routine
- ESLint authority: `eslint-config-vtex` via `.eslintrc`. Plugin: `eslint-plugin-jest`. Do not introduce competing configs
- Prettier authority: `.prettierrc`. Do not introduce alternate formatting configs
- Husky pre-push hooks run lint and tests on changed files (skipping on `beta`). Do not bypass with `--no-verify`

## Testing requirements

- Unit tests run on **Jest 22** via `yarn test`. Tests live in `tests/`, named `<name>.test.js`
- New code in `src/` **must include a test** in `tests/` covering at least the happy path and one failure case
- Bug fixes ship with a failing regression test before the fix
- Mock `@vtex/estimate-calculator` at the test boundary — do not let real estimate calculations run inside helper tests
- Snapshot tests are discouraged; use explicit assertions
- No coverage threshold is enforced. Do not add one unilaterally

## Versioning & releases

- **Releases are gated by Releasy** (`_releasy.json`)
- Agents do **not** bump the version in `package.json` or run `npm publish` / `yarn publish`
- `CHANGELOG.md` follows Keep a Changelog. Breaking changes go under `### Breaking Changes` — not under `### Changed`
- The `prepublishOnly` hook ensures `dist/` is rebuilt before publishing

## Security requirements

- This library does not handle PII directly — but its outputs feed UIs that do. Do not log card numbers, full CPF/CNPJ, full address, or email body inside helper functions
- Secrets must never appear in source

## What agents cannot change without human approval

- The public API surface (`src/index.js` re-exports, parameter signatures, return shapes, default values)
- `rollup.config.js` build configuration
- `package.json` `main`, `module`, or `dependencies` fields
- Major versions of devDependencies (Babel 6, Jest 22, ESLint 4, Rollup 0.56)
- `.eslintrc`, `.babelrc`, `.prettierrc` toolchain configs
- `_releasy.json`, `pachamama.config`, `.github/CODEOWNERS`
- Repository version (`package.json#version`) — Releasy manages it
- `npm publish` / `yarn publish` operations
- Adding CI/CD workflows under `.github/workflows/`
- Wholesale conversion of the codebase to TypeScript

## Governance

This constitution supersedes any individual preference or style guide in the repository. When a rule here conflicts with `AGENTS.md`, this document wins. Amendments require a PR that updates this file, with reviewers from the delivery-packages owners. The PR description must justify why the rule is now permanent rather than situational.
